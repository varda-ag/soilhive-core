# DAI: per-feature aggregates precomputed at ingestion, live aggregation only for filtered requests

The `/dai` endpoint was the heaviest CPU consumer on the database: for every viewport (per bbox, per zoom) a cache miss inserted an ephemeral UserGeometry (firing the `ST_MakeValid` + `ST_Subdivide` trigger), ran a `CROSS JOIN LATERAL` with four `COUNT(DISTINCT …)` aggregates **once per feature intersecting the AOI**, and deleted the geometry again — a write plus a per-feature aggregation on every GET.

The four per-feature counts (`num_soil_properties`, `num_props_below_30`, `num_dated_layers`, `num_distinct_years`) depend only on `dataset_layers`/`layers` rows and the `datasets` columns `deleted_at`/`gis_datatype` — data that changes only on ingestion and dataset mutations. The AOI only selects *which* features to return. So the per-feature rows are precomputed into `feature_dai_stats` (`feature_id` PK with `ON DELETE CASCADE` to `features`, `centroid geometry(Point,4326)` GiST-indexed, four integer counts), and the **unfiltered** `/dai` request becomes a GiST point lookup with no write churn. The H3 binning stays in the service layer, and the `dai:{filterId}:{bbox}:{resolution}` result cache (ADR 0008) stays in front — misses just become cheap.

- **The refresh SQL lives in the application** (`refreshDaiStats`, `data-layer/DaiStats.ts`), not in a DB function: formula changes ship with an app deploy, with no coordinated DDL on live databases. It must stay in lockstep with the unfiltered branch of `SoilDataStorage.getDaiPointData`: same predicates (`deleted_at IS NULL`, `gis_datatype != 'raster'`, deliberately **no** status filter), same counts. No `datasetIds` rebuilds everything; a dataset-id array recomputes rows for features *currently* linked to those datasets. Each mode is a **single statement** (upsert + prune via data-modifying CTEs): atomic on its own, and the full rebuild upserts in place so readers never observe an emptied table. The DDL is mirrored for already-migrated DBs by `backend/scripts/dai-stats-schema.sql`, and the full rebuild is duplicated once in `dai-stats-backfill.sql` — the price of not having a DB function is keeping that script in sync.
- **Filtered requests keep the live path.** Every `FilterCriteria` key changes what the counts include (explicit nulls are filters too), so `isUnfilteredDaiParameters` (`data-layer/DaiStats.ts`) routes to the precomputed table only when no criterion is present (`raster_filters` with only empty arrays counts as absent, matching `buildRasterSql`). Anything filtered runs the original ephemeral-geometry aggregation unchanged.

## Consequences

- **Staleness contract, same shape as ADR 0008:** every write path that changes DAI inputs must call `refreshDaiStats` alongside its `bumpCacheEpoch()`. Hooked: `runJob`'s `finally` for `BULK_LOAD` (scoped by the job's dataset; runs even on failure since partial batches may have committed), `DatasetService.deleteDataset` (scoped; **must run on soft-delete while `dataset_layers` rows still reference the dataset** — the bulk-delete job hard-deletes them right after), `DatasetService.updateDataset` when `gis_datatype` is touched, and `postSoilData`. `deleteFile` needs no hook (it never touches `dataset_layers`). A new write path that mutates `dataset_layers`/`layers` or dataset `deleted_at`/`gis_datatype` must refresh too.
- **Scoped refreshes resolve features through current `dataset_layers` rows.** After a hard delete that unlinks features from a dataset, a scoped refresh cannot see them anymore; hard-deleted features themselves are cleaned by the FK cascade, and the soft-delete-first ordering covers the bulk-delete flow. If a future write path hard-deletes links without a prior soft-delete refresh, it must do a full refresh (`refreshDaiStats(em)` with no ids).
- **Selection semantics changed at AOI edges:** features are selected by centroid-in-AOI instead of polygon-intersects-AOI. Point features are identical; a polygonal feature straddling the viewport edge with its centroid outside is now dropped, where before it was returned binned to a cell outside the viewport anyway.
- Storage is ~200 bytes per feature all-in (heap + PK + GiST) — noise next to `observations`; the table is rebuilt in one pass by the backfill script (minutes on production data, single transaction).
- Ingestion pays a scoped refresh (seconds, background job) so reads stop paying per-viewport aggregation.

## Considered options

- **Per-H3-cell precomputation per resolution** — faster still, but multiplies storage by resolutions (0–15), freezes the binning/scoring in the table, and the per-feature point scan is already cheap; binning in the service keeps the formula in one place.
- **Materialized view with `REFRESH … CONCURRENTLY`** — forces a full rebuild on every ingest; the plain table allows dataset-scoped incremental refreshes, which matches how data actually arrives.
- **Extending the precompute to filtered requests** — the counts are not decomposable per filter dimension (`COUNT(DISTINCT)` across datasets/layers); supporting arbitrary criteria would require keeping rows near the granularity of `dataset_layers` itself, erasing the gain.
