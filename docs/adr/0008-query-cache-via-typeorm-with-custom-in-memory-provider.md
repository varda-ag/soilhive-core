# Query-result caching: in-memory LRU behind TypeORM's cache, a raw-SQL wrapper, and one service-boundary key

To avoid re-running expensive spatial aggregate queries (DAI, filter coverage, soil-data counts) on every request, we cache **query results in memory**, not HTTP responses. One shared `lru-cache` store (`utils/query-cache.ts`, max 1024 entries, LRU eviction, per-entry TTL not refreshed on read) is reached through three deliberate paths, because the codebase's queries come in three shapes:

1. **TypeORM's opt-in query cache** (`.cache(ttl)` / `find({ cache: ttl })`) backed by a **custom in-memory provider**, for QueryBuilder-based queries (`filterRaster` internals, raster layer count, reference-data lists). The custom provider is not optional: TypeORM's default `"database"` store writes through the request's transactional query runner, and `transactionMiddleware` **rolls back** GET transactions — entries would never persist, and hits would still round-trip to Postgres. TypeORM derives the key from SQL + bound parameters automatically; never pass a fixed id to `.cache(id, ttl)`, which defeats that derivation and collides across inputs. Note TypeORM JSON-stringifies results before handing them to any provider and parses on every hit — this round-trip is baked into `SelectQueryBuilder` and was accepted; it also means hits on this path return fresh objects.
2. **`cachedQuery(entityManager, sql, params, ttl)`**, for the expensive queries that are deliberately **raw SQL** (`filterVector`, `getRasterCoverage`, `getSoilDataCount` — written raw for planner control; see comments in `SoilDataStorage`). TypeORM's cache cannot see `entityManager.query()` calls at all. The wrapper derives its key mechanically from the SQL text + `JSON.stringify(params)` — the same guarantee as TypeORM's keying, no hand-built keys. Results are stored by reference (no serialization); call sites must not mutate returned rows.
3. **One manual service-boundary key for the DAI** (`dai:{filterId}:{bbox}:{resolution}` in `FilterService.getDai`). Query-level caching is structurally impossible there: each computation inserts and then deletes an ephemeral UserGeometry, so the SQL parameters contain a UUID that never repeats. The manual key is complete because a Filter's content is immutable per id (dedup by content identity, ADR 0007), `/dai` carries no entitlement requirement, and `getFilterById` performs no ownership check. A hit also skips the per-request geometry insert/delete churn.

## Consequences

- **Invalidation: DB-signalled epoch on every write path, TTL as backstop.** The cache is per-process and the API runs multiple nodes, but all nodes share Postgres — so writes are signalled through it. A single-row `cache_epoch` version counter is bumped by **every write path that feeds cached queries**: the data-mutating pg-boss queues (`BULK_LOAD`, `BULK_DELETE`, in `runJob`'s `finally` — a failed bulk job may have committed partial batches) and the synchronous REST mutations (`updateDataset`/`deleteDataset` — including publish, `postSoilData` direct ingestion, `deleteFile` — raster queries join on `files.deleted_at`, and the vocabulary/license/procedure creates). Every node polls the counter every 30s (`utils/cache-epoch.ts`) and clears its local cache when the version changes, so post-write staleness is bounded by the poll interval on every node, not the TTL. **A new write path that affects cached results must call `bumpCacheEpoch()`.** The epoch was chosen over `LISTEN/NOTIFY` because it is state rather than an event: a node that was down just reads the current value on its next poll, and no dedicated listening connection outside TypeORM's pool is needed. A bare local `clear()` (without the shared counter) would be wrong — it only reaches the node that ran the write.
- **Pre-commit race, deliberately bounded by the poll.** REST writes bump mid-transaction (`transactionMiddleware` commits after the response), so a concurrent request can re-cache pre-commit data right after the local clear. `bumpCacheEpoch` therefore does not advance the node's own `knownVersion`: its next poll sees the new version and clears once more, after the commit. The race window is the poll interval, not the TTL.
- **TTLs are a pure backstop, not the freshness bound.** Both tiers sit at 12h (`CACHE_TTL_SPATIAL_MS`, `CACHE_TTL_REFERENCE_MS` in `utils/query-cache.ts`) — safe only because the epoch covers every known write path; they exist for watcher/poll failures (poll errors degrade gracefully to TTL expiry). If a write path is ever added without a bump, staleness runs to 12h — hence the rule above.
- **Deliberately not cached:** `/soil-data` feature queries (large GeoJSON payloads would evict everything else); single-entity slug lookups via `getEntity` (cheap indexed reads, and `/vocabulary/{id}` 301 slug-history behavior); `/config` reads (trivial PK lookups where admin read-your-writes matters); `/epsg` and `/procedures/techniques` (static in-process data, no DB query).
- Killswitch: `QUERY_CACHE_ENABLED=false` env var (default on; forced off under jest via `isJest()`), which also disables the DataSource cache config so all `.cache()` calls become no-ops. The provider logs hit/miss/size stats every 5 minutes.
- Expensive queries must stay on QueryBuilder or go through `cachedQuery`; a bare `entityManager.query()` is never cached.

## Considered options

- **HTTP response caching (apicache or middleware)** — rejected: routing is spec-driven (`express-openapi-validator`) with no per-route attach point, `transactionMiddleware` already monkeypatches `res.write`/`res.end` (double-wrapping risk), entitlement-gated responses would force per-user keys that prevent sharing identical computations across users, and `/vocabulary/{id}` 301 redirects would need special-casing.
- **TypeORM cache only** — the original plan; abandoned when implementation revealed the heaviest queries are raw SQL (invisible to it) and the DAI's ephemeral-geometry parameters can never produce a repeating key.
- **Hand-rolled memoization everywhere** — rejected: manual cache-key construction must remember every varying input; a missed input is a cross-user data leak. Manual keying is confined to the single documented DAI case.
- **TypeORM's default `"database"` store** — broken here, see above.
- **Redis store** — a shared cache with real invalidation, but new infrastructure; the `cache_epoch` counter delivers the cross-node invalidation half using the Postgres we already have. Revisit only if a shared (rather than per-node) cache itself becomes necessary.
- **`LISTEN/NOTIFY` instead of polling** — near-instant invalidation, but requires a raw connection outside TypeORM's pool, reconnect handling, and an epoch re-check after disconnects anyway; ~30s propagation is acceptable for soil-data ingests.
