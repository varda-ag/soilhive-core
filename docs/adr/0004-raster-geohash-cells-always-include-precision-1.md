# Raster geohash_cells always start from precision 1

All raster layers store geohash cells at every precision level from 1 up to their diagonal-selected maximum, rather than starting from a coarser level. The spatial filter query uses `&&` (exact array overlap) against `inputHashes` generated by `geohashesForGeometry`, which starts from precision 1 but stops adding levels once the bbox cell count exceeds 500. For large AOIs this means `inputHashes` contains only precision-1 cells. Any raster that was ingested without precision-1 cells — previously those with diagonal ≤ 50° — would fail the `&&` check entirely despite true spatial overlap. Storing precision 1 for every raster guarantees the two arrays always share at least one cell, and the subsequent `ST_Intersects` footprint check eliminates false positives cheaply.

## Considered Options

- **Prefix matching in SQL**: replace `&&` with `LIKE ih || '%' OR ih LIKE gc || '%'` — rejected because it cannot use the GIN index and scales poorly.
- **Expand inputHashes with all child hashes**: at query time, for each coarse hash generate all descendant hashes — rejected because at precision 5 a single precision-1 cell has ~131,000 children, making the query parameter unbounded.
